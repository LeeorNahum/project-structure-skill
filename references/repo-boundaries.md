# Repo Boundaries

Git is not the same decision as GitHub.

Most folders should have **no Git at all**. Do not initialize Git by default just because a folder contains files or because the parent workspace is Git-controlled.

## Default Git Posture

| Folder kind | Git | Remote |
|-------------|-----|--------|
| Surface folders (`Brand/`, `Planning/`, `Research/`, `Devices/`) | No by default | No |
| Atomic scoped folders (`<Project> Icon/`, `Project-Plan/`, `project-website/`) | Only when justified | Only if user requests |
| Workspace container root | Local Git for control plane | Rare; ask first |
| Scratch, imports, exports, temp work | No | No |

Local Git outside the workspace control plane is **uncommon**. Use it only when history, collaboration, releases, submodules, or explicit user request clearly need it.

Add a remote only when the folder has a durable identity worth syncing, sharing, deploying, releasing, or backing up as a standalone project.

## Canonical Naming Rule

Default to strict canonical naming:

- local workspace root: `Project Name`
- surface folder: `Brand/`, `Planning/`, `Web/`, `Firmware/` — organizational, usually not Git repos
- atomic publishable folder: `<Project> <Surface>` or `<Project> <AssetType>` — e.g. `Audiote Icon/`, `Audiote-Plan/`, `Audiote Website/`
- standalone code/package repo: `project-surface` or the package's actual deployable slug
- skill repo: `<skill-name>` or `<skill-name>-skill`, with the skill `name` matching the installed folder

Folders without the canonical project identity in the name are usually local-only, scratch, imported, generated, or not ready for GitHub.

A surface folder and an atomic folder are different:

- `Brand/` organizes work locally; it is not a remote candidate.
- `Brand/Assets/Audiote Icon/` is atomic, standalone in scope, and may become a repo if requested.

Exceptions are allowed, but ask before making them. A good exception has a clear public/package identity that would be weakened by forcing the parent project name into it.

## Remote Criteria

Suggest a GitHub remote only when at least one is true:

- the folder is an atomic canonical source for a product, package, firmware, device design, skill, app, library, brand asset set, release artifact, or durable public/private project
- the folder name is specific enough to stand alone as a repository title
- the folder needs collaboration, CI, releases, deployment, issue tracking, or backup across machines
- the user explicitly requested a remote for that folder

Do not suggest a remote merely because:

- the folder contains files
- the parent workspace is Git-initialized
- the folder lives under a canonical project workspace
- the agent wants a place to commit temporary work
- the folder was generated by a tool
- the folder is a surface organizer like `Brand/` or `Planning/`
- the folder is a one-off class/task/export unless the deliverable itself is the project

## Initialization Questions

Before initializing Git in a subfolder, decide:

```text
Is this folder an atomic publishable unit, a local-only surface, or part of a larger project root?
```

Before adding a remote, ask:

```text
Should this repo exist on GitHub under this exact name, and should it be public or private?
```

If the answer is unclear, keep it local with no Git.

## Nested Repos

A workspace container may hold nested repos. Keep the boundaries explicit:

- parent control-plane repo tracks root docs, submodule gitlinks, and shared standards
- child repos track their own source, docs, and release artifacts
- parent repo should not accidentally track child repo files
- child repos should not depend on untracked parent-only files unless documented

When a child repo must stand alone, give it its own `AGENTS.md` and install or document the skills it needs.

## Naming Smells

Pause before publishing names that are based on temporary process state, course/task context, tool residue, or date stamps.

Do not date-stamp main project directories, surface folders, or their direct subdirectories. Date suffixes belong in **Artifact Snapshot** folders when a dated deliverable packet is intentionally preserved — not in canonical asset, code, or planning folder names.

Ask whether the folder is a product/repo or just a moment in the process.
